# B2B 电商平台技术方案

> 依据《B2B操作手册-功能清单》整理，建设范围：**运营端**、**供应商端**、**PC 经销商端（PC 商城）**。  
> **不在本期范围**：业务端（业务员 H5）、经销商移动端商城（若手册中的 H5 站点仅作运营配置可保留为「PC 前台可配置项」，不单独开发业务员 App）。

---

## 1. 建设目标与范围

### 1.1 业务定位

- 平台作为 **B2B 订货与结算中枢**：经销商在 PC 端选品下单；供应商维护商品与发货；运营端负责会员、商品池、审核、营销、财务结算与系统配置。
- 与手册中 DMS 场景一致：**多仓库、渠道价、包装标签、企业/零售会员、子账号、授信与预存款、订单多级审核、供应商结算** 等。

### 1.2 三端职责边界

| 端 | 用户 | 核心职责 |
|----|------|----------|
| **运营端** | 平台管理员、运营、财务 | 站点与内容、会员与标签、类目与商品主数据、商品池与上架、全链路订单/支付/发货/售后/退款、营销、物流主数据、供应商入驻与账号、审核中心、财务报表、组织权限、系统设置 |
| **供应商端** | 供应商 | 商品发布与送审、素材与图片、订单发货、售后配合、结算与发票、运费模板、子账号与角色、运营报表 |
| **PC 经销商端** | 零售/企业会员 | 注册登录、企业认证、商城浏览下单、购物车与批量下单、支付（含线下凭证）、订单与售后、子账号与企业成员、预存款/授信、发票与地址、优惠券与评价 |

### 1.3 明确剔除（本期不做）

- **业务端（业务员 H5）**：线索、代客下单、订单催收、客户对账等 — 若未来需要，可复用同一套订单/会员 API 另起前端。
- **供应商/经销商独立 App**：除非合同要求，否则以 **响应式 Web** 或后续再立项。

## 2. 总体架构

### 2.1 逻辑架构

```

                    │            CDN / 静态资源 / OSS          │
                    └─────────────────────────────────────────┘

     │ (Vue/React)  │  │ (Vue/React)  │  │ (Vue/React)   │
     └──────┬───────┘  └──────┬───────┘  └──────┬───────┘

                             │ HTTPS / JSON

            └────────────────┼────────────────┘

                    ┌────────▼────────┐
```

**推荐形态**（与手册 Java 版一致）：**Spring Boot 单体或模块化单体** 起步，按包划分 `admin`、`supplier`、`mall`、`common`，数据库共享、事务边界清晰；流量增大后再拆 **订单 / 结算** 等核心服务。

### 2.2 部署架构（建议）

- **容器化**：Docker + Kubernetes 或 Docker Compose（中小规模）。
- **环境**：开发 / 测试 / 预发 / 生产；配置外置（Nacos / Spring Cloud Config / ENV）。
- **文件**：对象存储（OSS/COS/MinIO）存商品图、凭证、证照；图片处理（缩略图）与 CDN。
- **日志与监控**：集中日志（ELK/Loki）、链路追踪（可选 SkyWalking）、指标（Prometheus + Grafana）。

## 3. 技术选型建议

### 3.1 后端（Java）

| 类别 | 建议 | 说明 |
|------|------|------|
| 框架 | Spring Boot 3.x + Spring Security | 成熟、与手册技术栈一致 |
| API | REST；重要回调可考虑幂等设计 | 对外可预留 OpenAPI |
| ORM | MyBatis-Plus 或 JPA | B2B 复杂查询多，MyBatis 更灵活 |
| 校验 | Jakarta Validation | 参数与业务规则分层 |
| 任务 | Spring Scheduler + 队列（RocketMQ/RabbitMQ） | 账单、库存同步、消息推送 |
| 工作流（可选） | Flowable / 自研状态机 | 订单/售后/审核状态多 |

### 3.2 前端（三端）

| 端 | 建议 | 说明 |
|----|------|------|
| 运营 / 供应商 | Vue 3 + Element Plus / Ant Design Vue | 表格、表单、权限菜单成熟 |
| PC 商城 | Vue 3 / React + 自有或商用 UI | 侧重商品展示与下单体验、SEO 如需可考虑 SSR（Nuxt） |

构建：Vite；状态管理：Pinia；请求：Axios + 统一错误与 Token 刷新。

### 3.3 数据与中间件

- **数据库**：MySQL 8 / PostgreSQL；核心表读写分离与分库分表 **按需后置**。
- **缓存**：Redis（会话、热点商品、限流、分布式锁）。
- **搜索**（可选二期）：Elasticsearch/OpenSearch，用于商品与订单复杂检索。
- **定时任务**：XXL-JOB / PowerJob 或与 MQ 延迟消息结合。

## 4. 核心领域模型（概要）

以下为逻辑实体，实施时需落 ER 图与表结构。

- **组织与用户**：平台操作员、角色、权限资源、组织架构；供应商账号、子账号、角色。
- **会员体系**：零售会员、企业会员、会员等级、标签（渠道/包装）、归属组织、业务员关联、仓库绑定。
- **企业**：企业信息、证照、银行账户、认证审核记录。
- **商品**：分类（三级）、品牌、属性、规格、包装、SPU/SKU、交期、系列、渠道标签、素材库、自定义 Tab、凑单标记。
- **商品归属**：供应商商品池、平台上架/下架、审核状态、平台侧商品扩展（标签、推荐）。
- **价格与营销**：渠道价、会员价/等级价、满减满折满赠、优惠券（领取与使用规则）。
- **库存**：多仓库、可用/冻结/安全库存、库存流水；与订单锁定解耦设计。
- **订单**：主单、子单/发货维度、状态机（待审核→待支付→待发货→待收货→已完成/关闭）、定制单扩展。
- **支付**：支付单、支付方式、线下凭证、预存款、授信支付、支付密码策略。
- **物流**：物流公司、运费模板、发货单、物流轨迹（对接快递 100 等）。
- **售后**：售后单、退款申请、退款单；与供应商审核、平台退款衔接。
- **结算**：客户账单、供应商账单/汇总、结算明细、发票（客户/供应商维度）。
- **授信**：额度账户、账单日/还款、冻结规则、滞纳金（按产品配置）。
- **内容**：文章栏目/文章、站点装修 JSON（PC 首页楼层等）、帮助中心。
- **审计**：操作日志、登录日志、短信日志、API 日志（敏感字段脱敏）。

## 5. 关键业务流程（技术要点）

### 5.1 商品上架链路

供应商 **发布 → 提交审核 → 运营审核 → 运营上架入池** → PC 端可见（需叠加 **渠道标签、仓库、会员可见性** 过滤）。

技术要点：状态枚举统一；审核记录表；变更需版本或快照以便追溯；上架写缓存失效。

### 5.2 下单与价格

购物车合并 **包装单位换算**；结算时计算：**基础价 × 渠道/等级/倍率**、优惠券、活动（满减满折满赠）、运费（多模板合并规则）。

技术要点：计价服务独立方法便于单测；大促时热点 SKU 走缓存；下单 **幂等键**（防重复提交）。

### 5.3 订单审核（可配置）

平台开关：**下单是否需审核**；企业子账号 **是否需主账号审批**；与手册「双次审核」一致处用 **审核步骤配置表** 驱动，避免硬编码。

### 5.4 支付与预存款/授信

- 在线支付：对接微信/支付宝（服务商模式视资质而定）。
- 预存款：充值生成流水 → 可选 **运营审核** → 加余额；支付扣减需 **事务 + 乐观锁**。
- 授信：额度校验；月账单定时任务生成；还款与冻结用状态机；线下还款走 **转账审核**。

### 5.5 售后与退款

售后类型驱动不同状态流；**平台退款**与 **供应商审核通过** 节点用事件或表状态串联，保证资金与库存回滚一致（可 Saga 或本地消息表）。

### 5.6 结算与对账

按 **账期/订单完成** 生成结算明细；供应商与客户侧分开汇总；导出 Excel 可走异步任务 + 文件 URL 下载，避免超时。

## 6. 接口与权限设计

### 6.1 API 划分

- `/api/admin/**`：运营端，JWT + RBAC，细粒度到菜单/按钮。
- `/api/supplier/**`：供应商端，`supplier_id` 数据隔离。
- `/api/mall/**`：PC 商城，会员 Token；部分接口匿名（商品浏览按运营配置）。

### 6.2 多租户与隔离

- 供应商数据：**supplier_id** 强制过滤（MyBatis 拦截器或 Repository 层约定）。
- 经销商：**member_id / org_id**；子账号继承主账号企业上下文。

### 6.3 对外开放（按需）

- 内部 ERP/U9：预留 **同步视图或 MQ 事件**（手册提到操作员可来自 U9 同步）。
- 物流查询、短信邮件：**适配器模式** 接第三方。

## 7. 非功能需求

| 维度 | 建议 |
|------|------|
| 安全 | HTTPS、密码 BCrypt、支付密码独立、敏感字段加密、CSRF 防护（Cookie 场景）、接口限流、防刷验证码 |
| 审计 | 关键操作留痕；财务与审核操作不可物理删除，用逻辑删除 + 变更日志 |
| 性能 | 列表分页、慢查询治理；报表走汇总表或异步预计算 |
| 可用性 | 数据库备份、MQ 堆积告警；发布灰度（可选） |

## 8. 分期实施建议（里程碑）

**阶段 A — MVP（可演示交易闭环）**  
运营：登录、类目品牌、会员（简化）、供应商入驻与账号、商品池与上架、订单列表与详情、基础支付（可先一种）、发货、基础报表。  
供应商：商品发布与审核流、订单发货。  
PC：注册登录、商品列表详情、购物车、下单、订单列表、一种支付方式。

**阶段 B — B2B 完整能力**  
渠道/包装标签、多仓库库存、子账号与审批、预存款与线下凭证审核、售后退款、优惠券与满减满折、运费模板、发票与结算账单导出、企业认证两级审核、授信（按优先级拆分）。

**阶段 C — 增强**  
可视化装修（站点配置）、商品问答与到货通知、复杂报表与 ES 搜索、API 开放与对接 ERP。

## 10. 风险与依赖

- **支付与发票**：依赖商户资质与税控政策，需尽早确认对接方式。
- **第三方物流**：轨迹查询费用与稳定性。
- **历史系统**：若必须对接 U9/ERP，接口字段与主数据（客户、物料）需一期拉齐。
- **权限复杂度**：运营端菜单深，建议 **权限元数据化**（资源表 + 角色绑定），避免前后端硬编码分叉过多。
